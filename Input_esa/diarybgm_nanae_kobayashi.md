## DIARY BGMってなに？

日記を書くと、その日の気分に合ったBGMが自動で生成されるWebアプリです。

**使い方はシンプルに3ステップ**
1. Googleアカウントでログイン
2. 今日あったことや気持ちを自由に書く
3. AIが感情を読み取って、ぴったりのBGMを生成してくれる

過去の日記も振り返れて、そのときのBGMも一緒に再生できます。

## なぜこのアイデアを作ったか

日記って、続かないですよね。

続かない原因はいくつかあると思っています。書いた後に価値を感じられない。一生懸命書いても誰からもフィードバックがないのでつまらない。振り返ることで内省が深まるというけど、長い文章を読み返すのが正直めんどくさい。

そこで考えたのが、**「書いた後に振り返りたくなる体験」** を作れないか、ということです。

DIARY BGMでは、自分の書いた言葉をもとに音楽が生成されます。長い文章を読み返してその時の気持ちを思い出すのではなく、「音」という新しい情報が加わることで、その時の気持ちをより自然に思い返せるようになります。

おまけの効果として・・・たとえば怒っていたり落ち込んでいる時に日記を書くと、思っているより壮大な曲が流れたりします。「こんなに怒ってないわw」と、重い気持ちが少し軽くなる瞬間があって、それがわりと気に入っています。


## Claude Codeで作ってみた

### どんなふうに進めたか

「こういう画面を作りたい」「ここのデザインをこう変えて」——基本的にはこれだけです。

コードは一行も自分で書いていません。スクリーンショットを見せて「もう少しレコードを大きく」と言えばそう直してくれるし、エラーが出たらそのまま貼り付ければ原因を調べて直してくれます。

Figmaを使ったのは、カラーパレットを考えたり、ログイン画面のUIをイメージするくらい。それ以外のほとんどはClaude Codeとの会話で進みました。

### 使った技術とサービス

| サービス | 何をしているか |
|---|---|
| **Gemini 2.5 Flash** | 日記のテキストを読んで感情を分析し、BGM生成のためのプロンプトを作る |
| **fal.ai** | Geminiが作ったプロンプトをもとに、実際の音楽ファイル（20秒）を生成する |
| **Supabase** | 日記データと生成された音楽ファイルを保存する |
| **NextAuth + Google** | Googleアカウントでログインできるようにする |
| **Vercel** | 作ったアプリをインターネットに公開する |

### 仕組みの流れ

```
日記を書く
  ↓
Gemini 2.5 Flash が感情を分析 → BGMのプロンプトを生成
  ↓
fal.ai が実際の音楽ファイルを生成（20秒）
  ↓
Supabase に日記データと音楽ファイルを保存
  ↓
ブラウザで再生
```

### 感情 × カラー × BGMの対応

Geminiは日記を読んで、10種類の感情カテゴリのどれかに分類します。感情ごとにカラーとBGMの方向性があらかじめ決まっていて、それをもとに音楽が生成されます。

| 感情 | カラー | BGMの雰囲気 |
|---|---|---|
| 燃えている | 🟠 オレンジ | パワフルなオーケストラ・激しいビート |
| 楽しい | 🟡 イエロー | 明るいアコースティックギター |
| 穏やか | 🟢 グリーン | 自然音を交えた穏やかなアンビエント |
| 満たされた | 🔵 ブルー | 柔らかいアンビエントピアノ |
| ぼんやり | 🟣 ライラック | lo-fiヒップホップ・ゆるいビート |
| 疲れた | 🟤 ブラウン | スローなミニマルピアノ |
| もやもや | 🩶 グレイッシュ | 曇った雰囲気のインディーフォーク |
| 寂しい | 🩵 スチールブルー | メランコリックなソロピアノ |
| 悲しい | 💧 ダークティール | 悲しい弦楽器・静かな音 |
| 怒り・焦り | 🔴 ラズベリーレッド | 緊張感のある弦楽器・不協和音 |

カラーはUIにもそのまま使われていて、日記を書いた日の感情がアプリ全体の色調に反映されます。

### 「これはすごい」と思った瞬間

驚いたのは2つ。

1つは、**感情分析の精度が高いこと**。日記のテキストから感情の種類・強度・キーワードまで読み取って、それに合ったBGMを生成する——このロジックをどう組むか自分には見当もつかなかったけど、Claude Codeが適切なAPIを選んでそのまま実装してくれました。

もう1つは、**カラーパレットを画像で渡したらそのまま反映してくれたこと**。Pinterestで好みのカラーパレットを探してきて「この画像の色に合わせて」と伝えるだけで、UIに反映されました。
カラーコードを渡さなくてもよくて便利すぎた〜

### 日記の内容はDBで読めないようにした

社内に公開するにあたって、**日記の本文をデータベースレベルで暗号化**しました。

技術的には AES-256-GCM という暗号化方式を使っています。Supabaseのダッシュボードを直接開いても、保存されているのは暗号文だけ。私自身もDBを見ても日記の内容は読めません。アプリ上でのみ、本人がログインした状態で復号されて表示されます。

「社内に公開するなら、書いた本人が安心して使える状態にしたい」と考えてClaude Codeに相談したら、実装方針を提案してそのまま組み込んでくれました。

## 大変だったこと・躓いたこと

### 音楽生成AIの選定

BGMを生成するサービスの選定が一番の難所でした。

| 試したサービス | やめた理由 |
|---|---|
| HuggingFace MusicGen | 生成が遅すぎてUXに耐えられなかった |
| Stability AI | 品質はよかったが有料プランの壁があった |
| **fal.ai（現在）** | 速度・品質・コスト感のバランスがよかった |

「このAPIに切り替えて」と言えばコードはすぐ直してもらえる。でも**何を選ぶか・なぜ合わないかの判断**は自分でしないといけませんでした。

### fal.aiは事前チャージが必要だった

fal.aiは使った分だけ課金されるサービスですが、先にクレジットをチャージしておかないと使えません。実装が完成してから気づいて、チャージするまで動かせない時間が発生しました。**AIサービスを使う前に課金体系と事前準備を確認する**のは大事な教訓です。

### APIキーまわりのトラブルあれこれ

- HuggingFaceのAPIキーに全角の括弧が混入していてエラー（コピペミス）
- Supabaseの環境変数をVercelに設定し忘れていてデプロイ後に動かなかった
- Geminiのモデル名が変わっていて、古い情報のまま設定したら404エラー



## 現時点のプロダクト課題

- BGM生成に時間がかかる（30秒前後）。待ち時間のUXを改善したい
- 音楽の品質にばらつきがある。プロンプトの精度をもっと上げたい
- 日記を書くハードルがまだ高い。入力欄が白紙だと手が止まる


## 学び

こんなに複雑なロジックを挟んだアプリが、こんなに簡単に作れるとは思っていませんでした。

今回Figmaを触ったのは、カラーパレットを考えたりログイン画面のUIをイメージするくらい。実装はすべてClaude Codeとの会話で進みました。

**APIを使ったプロダクトを設計するとき、実装できるかより「費用対効果」が先に来る問題だと気づきました。**
無料枠の範囲でどこまで体験を作れるか、クレジットが尽きたら使えなくなるサービスをどう扱うか・・・これはビジネス的な問いで、AIが答えを出してくれるものではありませんでした。
作る技術の敷居が下がった分、設計の意思決定がより重要になっていると感じています。

そして正直、**この先はアイデア勝負の時代だと思いました。**

「作れるかどうか」ではなく「何を作るか」が問われる。デザイナーじゃなくても、エンジニアじゃなくても、アイデアと言語化さえできれば動くものが作れる時代が来ている。
そのことを、身をもって体験できた1週間でした。
危機感も感じましたが、とっても楽しくワクワクしました！プロダクト作りってやっぱり楽しい！！
